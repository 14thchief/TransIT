# SoftGATE Admin v2

> SoftGATE Admin (2024)

## Project Overview

SoftGate is multi-tenant payment gateway management system.

## Technologies Used

- **Frontend Framework**: React 18 with Vite
- **State Management**: Redux Toolkit
- **Form Handling**: React Hook Form, React Select
- **Icon Library**: React Icons
- **Styling**: Sass
- **API Management**: Redux ToolKit Query
- **Others**: React Tabs, React Daterange, React Datepicker, React-PapaParse

## Project Setup

1. Clone the repository:

   ```bash
   git clone https://labhub.softalliance.com/softgate/softgate_admin_v2
   cd softgate_admin_v2
   ```

2. Install dependencies:

   ```bash
   npm install
   ```

3. Start the development server:

   ```bash
   npm run dev
   ```

4. Build for production:

   ```bash
   npm run build
   ```

5. Preview production build:

   ```bash
   npm run preview
   ```

## Collaboration Guidelines

To ensure a smooth development process and maintain high code quality, follow the guidelines below for collaboration:

- **Branching Strategy**:

  - Use dedicated branches for new features, fixes, and other changes.
  - Naming conventions:
    - `feat/feature-name`: For feature development (e.g., `feat/add-calendar-component`).
    - `hotfix/issue-description`: For critical bug fixes (e.g., `hotfix/fix-login-bug`).
    - `chore/description`: For non-functional changes such as updating dependencies (e.g., `chore/update-libraries`).

- **Commit Messages**:

  - Follow conventional commit standards to make the commit history clean and informative.
  - Common prefixes to use:
    - `feat`: For adding new features.
    - `fix`: For fixing bugs.
    - `chore`: For routine tasks or changes that do not modify the functionality (e.g., refactoring).
    - `docs`: For changes related to documentation.
    - `style`: For formatting or styling changes that don't affect the code logic.
    - Example: `feat: add calendar component` or `fix: resolve login issue`.

- **Pull Requests**:

  - All pull requests (PRs) must be reviewed and approved by at least one other team member before merging into the main branch.
  - Ensure your PRs are focused on a single change or feature to keep reviews manageable.

- **Code Quality**:
  - Run **ESLint** and **Prettier** before committing to ensure code consistency and to catch linting errors.
  - Ensure there are no linting errors or warnings before submitting a pull request.
